Claude Code + Codex
異モデル批判レビューで質を上げる

Claude Code（Anthropic）の中から OpenAI Codex（GPT-5.4）を呼び出し、書いたコードを別のAIに批判させる。質の向上とトークン節約を両立するワークフローガイド。

なぜこれをやるのか

盲点の補完

同じモデルの限界を超える

同じモデルが書いてレビューすると、同じ思考パターンで見逃す。別モデルに批判させて盲点を突く。

トークン節約

Claudeのコンテキストを温存

レビューや調査をCodexに委譲。Claudeのコンテキストには結果サマリだけが載る。

並行作業

レビュー中も手を止めない

Codexをバックグラウンド実行しながら、Claude側で別の実装を進められる。

アーキテクチャ

Claude Code → 薄いブリッジ → Codex CLI の3層構造。Claudeのコンテキストには出力結果だけが返ります。

セットアップ

前提条件
Claude Code がインストール済み、Node.js (npm) が使える環境であること。

Codex CLIをインストール

npm install -g @openai/codex

OpenAI認証

codex login

Claude Code内で状態確認

/codex:setup

Codex CLIの状態・認証状況を自動チェック。未インストールの場合はその場でインストールも可能。

コマンド一覧

コマンド 用途 ファイル変更
/codex:review gitの変更差分をCodexにコードレビューさせる 読み取りのみ
/codex:adversarial-review 設計判断そのものを批判する敵対的レビュー 読み取りのみ
/codex:rescue 調査・修正・デバッグをCodexに委譲 書き込み可
/codex:setup Codex CLIの導入・認証確認 —

/codex:review

/codex:review # ワーキングツリーの変更をレビュー /codex:review --background # バックグラウンドで実行 /codex:review --base main # mainからの差分をレビュー /codex:review --scope branch # ブランチ全体をレビュー

/codex:adversarial-review

/codex:adversarial-review # 全体を批判的にレビュー /codex:adversarial-review 認証周りが不安 # フォーカス指定 /codex:adversarial-review --background --base main # 組み合わせ

/codex:rescue

/codex:rescue このテストが落ちる原因を調べて修正して /codex:rescue --background 全エンドポイントのエラーハンドリングを見直して /codex:rescue --resume # 前回の続きから再開

実践ワークフロー

基本: 実装 → 批判レビュー → 修正

# 1. Claudeに実装させる あなた: 「ユーザー認証のミドルウェアを実装して」 Claude: （実装する） # 2. Codexに敵対的レビューさせる（バックグラウンド） あなた: /codex:adversarial-review --background （Codexレビュー中、Claudeで別作業を進める） # 3. 結果を確認 Codex: 「トークン検証でタイミング攻撃の可能性、 リフレッシュトークンのローテーションが未実装」 # 4. Claudeに修正させる あなた: 「Codexの指摘を踏まえて修正して」 Claude: （指摘箇所を修正）

応用: Claudeが詰まったらCodexに投げる

Claude: 「このエラーの原因が特定できません…」 あなた: /codex:rescue このエラーの根本原因を調べて修正して Codex: （独立コンテキストで調査・修正を実行） 「原因: DBマイグレーションのカラム名不一致。修正済み。」

adversarial-review の設計思想

レビュアーのスタンス
「この変更を出荷すべきでない理由を探せ」— 通常レビューが「問題がないか確認する」のに対し、adversarial-reviewは「問題があると仮定して証拠を探す」アプローチ。

攻撃対象（何を見るか）

出荷後に高コストになる障害を優先的に探します。

  • !認証・権限・テナント分離・信頼境界
  • !データ損失・破損・重複・不可逆な状態変更
  • !ロールバック安全性・リトライ・部分障害・冪等性
  • !レースコンディション・順序依存・古い状態・再入
  • !空状態・null・タイムアウト・依存サービスの劣化
  • !バージョン差異・スキーマ変更・マイグレーション
  • !障害を隠す/復旧を困難にするオブザーバビリティの欠落

報告基準（何を報告しないか）

報告される
具体的なコード行 + 現実的な障害シナリオ + 影響度 + 修正案がセットになったもの
報告されない
スタイルや命名の指摘、根拠のない推測、「念のため」レベルの懸念

トークン節約の効果

Claude単体で全部やる場合
ファイル読み + 実装 + レビュー再読み + レビュー思考 + 修正
Codexに委譲する場合
ファイル読み + 実装 + 結果受取 + 修正
仕組み
レビューの思考過程・ファイル読み込み・中間推論はすべてCodex側（OpenAI課金）で完結。Claudeのコンテキストには結果サマリだけが載るため、ファイル数が多いレビューほど差が開きます。

なぜ「異モデル」が効くのか

構造的限界

同一モデルの盲点

自分が書いたコードを自分でレビューする人間と同じ。同じ思考パターンで書いたものは、同じ思考パターンでは見逃しやすい。

視点の補完

異なる推論パターン

異なる学習データ・推論プロセスを持つモデルが「壊す側」を担当することで、Claudeの盲点をGPT-5.4が突く（逆もある）。

開発の基本原則

書く人 ≠ 壊す人

人間のチーム開発でも「作者≠レビュアー」は基本。モデル間の得意分野の違いが、レビューの網羅性を高める。

FAQ

OpenAI APIの料金は別途かかる？
はい。Codex CLIの利用はOpenAIの課金です。Claude CodeとCodexで別々のAPI課金になります。
Codex側がファイルを書き換えることはある？
/codex:rescue はデフォルトで書き込み可能（--write）です。/codex:review と /codex:adversarial-review はレビューのみで、ファイルを変更しません。
Claude Codeのプラグインはどこから入れる？
Claude Codeのプラグインマーケットプレイスからインストールできます。openai-codex プラグインを探してください。
レビュー結果をClaudeが勝手に修正し始めない？
修正しません。レビュー結果提示後は必ずユーザーに確認を取る設計になっています。どの指摘を修正するかはあなたが決めます。
